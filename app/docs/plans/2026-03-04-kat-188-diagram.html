1 — Overview

Scope

  • Task rows in the left panel match Mock 14 structure and interaction patterns
  • Task state updates reflect live run activity while the orchestrator is active
  • High-activity and no-activity row detail states match the provided detail mocks
  • Left-panel task states stay in sync with right-panel spec task states

Approach Selected

Main-owned task-activity projection + typed runtime snapshots. A typed task_activity_snapshot event emitted from main process. Renderer consumes snapshots and renders both panels from the same projected source.

Approaches Rejected

  • Renderer-only derivation from run/message events: brittle, no shared contract, hard to test
  • Polling-based sync via run:list + roster: laggy UX, IPC churn, race conditions

    O->>P: run_state_changed pending
    P->>P: Seed snapshot from spec tasks

    P->>P: Update active task detail
    P->>IPC: task_activity_snapshot

    P->>P: Settle in-progress items
    P->>IPC: terminal snapshot
    IPC->>R: Final state update

On Pending

Emit initial snapshot. Mark first unresolved task in_progress with high activity. Seed task list from applied spec document (or draft fallback).

On Message Update

Update top active task with one-line activity detail text. Emit fresh snapshot. Map agent progression to task row deterministically.

On Idle/Error

Settle in-progress items. Clear high-activity detail. Emit terminal snapshot. Preserve completion status from prior toggles.

Projection source priority: 1) Applied spec tasks, 2) Latest draft task list, 3) Empty snapshot. The projector never randomizes task selection; ordering is deterministic for stable tests.
5 — Panel Composition

Left Panel — TaskTrackingSection

Replaces preview-state task scaffolding in LeftStatusSection.

  • Header summary: 4 in progress · 3 done · 4 waiting
  • Status icon + title + disclosure chevron per row
  • activityLevel=none: compact row only
  • activityLevel=high: detail line + specialist badge

Right Panel — TaskList overlay

Keeps interactive checkbox surface. Enhances display from same snapshot.

  • Status badge reflects TaskTrackingStatus projection
  • Optional inline activity detail for high-activity rows
  • Checkbox toggles still delegate to useSpecDocument.toggleTask
  • Post-persist, local state merges into snapshot projection

AppShell Wiring

useIpcSessionConversation holds latest taskActivitySnapshot in conversation state. AppShell distributes the snapshot to both LeftPanel and RightPanel via props. ChatPanel exposes a callback onTaskActivitySnapshotChange for the runtime event flow.

6 — State Sync Contract
Data Writer Precedence
Task completion status specSave / specApplyDraft (checkbox) Authoritative
Activity detail, level, agent Runtime projector (main process) Layered on top
Task list ordering and titles Spec document (applied or draft) Source of truth
Conflict rule: If a user marks a task complete while runtime reports activity, completion status stays complete. Activity detail clears on the next snapshot. Persisted state always wins.

Resilience

  • Unmappable activity goes to a synthetic unmapped bucket (not rendered; logged)
  • Absent snapshot events: fall back to persisted spec task list with no-activity rendering
  • Unknown statuses treated as not_started

8 — Risks and Mitigations

Task ID Drift

IDs drift when markdown task order changes.

Mitigation: Deterministic title-based IDs + parser stability tests.

Event Ordering Races

Checkbox toggles and incoming runtime snapshots interleave.

Mitigation: Merge precedence with persisted status as authoritative.

Projector Scope Creep

Main-process projection complexity grows beyond ticket scope.

Mitigation: Single snapshot event, deterministic mapping, no extra IPC channels.

Non-goals: Full orchestrator rewrite, center-panel decision flow changes (KAT-187), left-panel tab architecture redesign, spec panel comments/threads.
